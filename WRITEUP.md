How about trying a hybrid search? We could combine a traditional keyword search engine like Elasticsearch with a semantic search. The idea is to first perform a keyword search to narrow down the results, and then apply semantic search on the filtered results to rank them based on relevance. This way, we can leverage the strengths of both approaches. I liked this idea, but the keyword search engine needs to be carefully tuned and assume that it narrows down the results effectively, so that the semantic search isn't to costly.

I liked the previous idea, the 2 stage arhitecture, but the individual components need to be improved. I thought what if we had an incoming query, we can send it to a LLM to map into the json format we have for companies. Then I embed the json for the company and compare it with cosine similarity to the embeddings in our database. Then we can set a threshold to pick for example the first 20 most similar companies. It addresses the fundamental asymmetry of vector search. Comparing a 5-word query ("Logistics in Germany") directly to a 500-word company profile often yields poor vector alignment. Why choose this design? By using an LLM to generate a synthetic, perfect-match JSON first, we force the query into the exact same "semantic shape" and vocabulary distribution as our database documents, vastly improving the baseline retrieval quality without having to run an LLM on every single company. This approach is optimized for speed, cost and simplicity, but it sacrifices logical precision for semantic proximity. This system will struggle with queries with hard constraints, numeric thresholds and logical exclusions. 

We have something that is fast, cheap and simple, but it has some limitations. Let's do even better. The weak spot is the usage of cosine similarity. With the cosine similarity, we can pull the top 100 most similar companies, but we don't have a good way to rank them. We can use a LLM to rerank the top 100 results. The LLM can be prompted with the query and the top 100 results, doing some prompt batching to speed up the process, and it can output a ranked list of the most relevant companies. This strips out the false positives that tricked the cosine math, leaving you with a clean Top 20.

Next I started implementing this idea. I got some decent results, but the more queries I tested, problems started to appear. At first the pipeline was leaning to much into the semantic meaning of the company description, so I decided instead of treating the entire company profile as a flat string that I should prepend field tags to the corpus. Furthermore, the pipeline could not be trusted to handle numbers, so I did add a deterministic pre-filtering step. Also removed the values used for the filters from reaching the NLP model, so that it doesn't get confused by them.

# Final design

The final design consists of a 4-stage pipeline. When a query comes in it goes through the following steps:
1. **Dynamic Intent Parsing & Stripping**: The system uses regex to extract hard quantitative and geographical rules. Crucially, it strips these values from the text to create a "clean semantic intent." This prevents the "semantic penalty," where NLP models assign low scores simply because a profile's text doesn't contain literal numbers.
2. **High-Recall Retrieval**: The clean semantic intent is embedded using `all-MiniLM-L6-v2`. A rapid cosine similarity search pulls the top 200 conceptual matches from the vector space.
3. **Deterministic Hard Filtering**: Before any expensive evaluation occurs, the system loops through the top 200 candidates and ruthlessly drops any that violate the extracted mathematical or geographic constraints.
4. **High-Precision Reranking**: The surviving candidates are passed to `bge-reranker-large`. Because it evaluates the query and document simultaneously, it catches complex B2B directional relationships (e.g., "supplier" vs. "buyer"). The model's raw logit outputs are converted into probabilities via a Sigmoid function, and a strict 60% confidence threshold is applied to discard borderline noise.

# An analysis of the performance trade-offs

I optimized heavily for a balance of accuracy, speed, and cost. By keeping the embedding and reranking models entirely local, API costs are zero and latency is kept to a minimum. Accuracy is maximized because the system assigns the right tool to the right job: deterministic regex for hard numbers, and a neural cross-encoder for fuzzy business relationships.

The main tradeoff I intentionally made was trading recall for precision. By enforcing strict pre-filters and a hard 60% confidence threshold, I intentionally built a strict system. I risk dropping a potentially relevant company if its data is too messy, but I make this trade to guarantee that the final Top 20 results are completely free of noise.

One assumption that I made was that the queries won't be filled with garbage. Hopefully the frontend would prevent the user from sending malicious queries, but if it does happen, the system will still return a Top 20, but it will be garbage. The system is designed to be robust to messy company data, but not to messy queries.

One more thing to note is that I used off-the-shelf models for the embedding and reranking. I didn't have the data to fine-tune them, but I did some prompt engineering to make them work better for my use case. I also used a different model for the embedding and reranking, because the embedding model is optimized for speed and the reranking model is optimized for accuracy.

# Potential errors that could arise

During the testing phase of the system, I noticed one issue. That the reranker model wasn't always very confident in its predictions. This could be due to the model being trained on a different distribution of data than what we are using. This could lead to issues considering that the 60% confidence threshold was set arbitrarily.

In my testing, the pipeline never returned a wrong choice with high confidence. Now, in a production environment, we could actually track the queries and the responses turned out by the system. With enough of these data we could hopefully fine-tune the reranker model to our specific distribution of data. This would allow us to have a more accurate confidence threshold and improve the overall performance of the system. 

# Scalability

To scale to 100,000 companies, I would replace the in-memory NumPy exact dot-product calculation with an Approximate Nearest Neighbor (ANN) index like HNSW using FAISS or a dedicated vector database like Milvus or Qdrant. Exact distance calculations become too slow at this scale, whereas ANN provides logarithmic retrieval times. Additionally, I would abandon Pandas and push the deterministic hard filters—like revenue and country code—directly into the vector database's metadata filtering layer. This ensures the system instantly prunes irrelevant records at the database level before executing the semantic vector search, keeping Stage 1 retrieval latency in the milliseconds regardless of dataset size.
